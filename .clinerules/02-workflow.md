# 工作流规范

## 规划阶段（Plan Mode）

收到新功能需求时：
1. **理解需求**：追问目标、范围、约束，直到完全理解
2. **探索代码**：使用 list_files、search_files、read_file 了解现有代码结构
3. **评估影响**：识别会影响哪些文件和模块
4. **制定计划**：将工作分解为原子任务（每个 2-5 分钟）
5. **确认计划**：将计划存到 `docs/plans/` 目录，等用户确认

## 执行阶段（Act Mode）

1. 按计划逐个任务执行
2. 每个任务遵循 TDD（见 testing.md）
3. 每完成一个任务，更新 task_progress
4. 每个有意义的变更后 git commit
5. 所有任务完成后使用 attempt_completion

## 计划文件格式

存放在 `docs/plans/` 目录，命名格式：`YYYYMMDD-feature-name.md`

每个计划包含：
- **目标描述**：清晰的目标陈述
- **影响的文件列表**：需要修改/创建的文件
- **原子任务清单**：带复选框的任务列表
- **验收标准**：如何验证完成
- **风险点**：可能的坑和注意事项

## 任务粒度原则

每个原子任务应该：
- 可以在 2-5 分钟内完成
- 有明确的验收标准
- 可以独立测试
- 完成后可以安全 commit

## 例外情况

以下情况可以跳过完整规划流程（需用户确认）：
- 简单的 typo 修复
- 配置文件微调
- 文档更新
- 已经非常明确的 bug 修复
