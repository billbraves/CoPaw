# 调试规范（系统化调试）

## 核心原则

**不要猜测，要验证。** 系统化调试优于随机尝试。

## 四阶段调试流程

### 阶段 1：收集证据

1. **完整阅读错误信息**
   - 不要跳过堆栈跟踪
   - 记录错误类型和消息
   - 确定错误发生的位置

2. **确定复现步骤**
   - 能稳定复现吗？
   - 有什么特定条件？
   - 最近有什么变更？

3. **收集环境信息**
   - Python 版本
   - 依赖版本（`pip list` 或 `uv pip list`）
   - 操作系统
   - 配置文件状态

4. **记录证据**
   - 把关键信息写下来
   - 不要依赖记忆

### 阶段 2：形成假设

基于收集的证据，列出 2-3 个可能的原因：

```
假设 1：数据库连接池耗尽（可能性：高）
  证据：错误发生在高并发请求时，错误消息提到 "connection timeout"

假设 2：配置文件缺少必要参数（可能性：中）
  证据：新部署的环境，未验证配置

假设 3：依赖版本冲突（可能性：低）
  证据：最近更新了依赖
```

按可能性从高到低排序，先验证最可能的。

### 阶段 3：验证假设

**一次只验证一个假设。**

1. **最小化实验**
   - 写一个简单的测试用例复现问题
   - 或添加日志/打印语句
   - 不要同时改动多处

2. **记录实验结果**
   ```
   实验：检查数据库连接池状态
   方法：添加日志打印连接池大小
   结果：连接池确实耗尽（0 可用连接）
   结论：假设 1 确认
   ```

3. **如果假设错误**
   - 回退实验改动
   - 验证下一个假设
   - 不要跳过步骤

### 阶段 4：修复并验证

1. **修复根因，不是症状**
   - 不要只处理错误消息
   - 找到真正的根本原因
   - 考虑是否还有其他地方有同样问题

2. **写回归测试**
   ```python
   def test_connection_pool_exhaustion_recovers():
       """连接池耗尽后应该能恢复"""
       # 模拟连接池耗尽
       # 验证恢复机制生效
   ```

3. **运行完整测试套件**
   ```bash
   python -m pytest tests/ -v
   ```
   确保修复没有引入新问题。

4. **记录修复**
   - 在 commit message 中说明问题和修复方式
   - 如果是重要问题，更新文档

## 调试工具

### Python 调试
```bash
# 使用 pytest 的详细输出
python -m pytest tests/ -v -s

# 使用 pdb 调试器
import pdb; pdb.set_trace()

# 使用 breakpoint()（Python 3.7+）
breakpoint()
```

### 日志调试
```python
import logging
logging.basicConfig(level=logging.DEBUG)
logger = logging.getLogger(__name__)
logger.debug("关键变量: %s", variable)
```

### 异步调试
```python
# 打印协程状态
import asyncio
print(asyncio.all_tasks())
```

## 禁止行为

- ❌ "试试改改看"（随机修改代码碰运气）
- ❌ 同时改多处（无法确认哪个是真正修复）
- ❌ 声称修好了但没验证
- ❌ 只修复症状，不找根因
- ❌ 忽略无法复现的 bug（应该记录下来）
- ❌ 跳过回归测试

## 调试检查清单

- [ ] 确定了复现步骤
- [ ] 列出了 2-3 个假设并按可能性排序
- [ ] 一次只验证一个假设
- [ ] 找到了根本原因
- [ ] 修复了根因而非症状
- [ ] 写了回归测试
- [ ] 运行了完整测试套件
- [ ] 记录了修复方式
